# 📌 二、LSM-tree 日志结构数据库

日志结构数据库（如 RocksDB、LevelDB）采用 **LSM-tree（Log Structured Merge Tree）** 结构。

---

# 1️⃣ 什么是“空间放大率低”？

空间放大率 = 实际占用空间 / 用户数据空间

“空间放大率低”意味着：

> 数据库不会因为内部机制而长期占用过多冗余空间。

---

## 为什么空间放大率较低？

### 1️⃣ 旧数据会被清理

* 数据从 MemTable 刷入 L0
* 逐级向下 Compaction
* 旧版本会被清除
* 不会永久重复占用空间

---

### 2️⃣ 冷数据可压缩

* SSTable 支持压缩
* Metadata 占比小
* 数据密度高

例如：

```
10 字节数据 + 5 字节元数据 ≈ 15 字节
```

保持较低空间膨胀。

---

# 2️⃣ 读放大（Read Amplification）

读放大：

> 一次逻辑读取 → 多次磁盘访问

---

## 原因

LSM 分层结构：

```
MemTable
↓
L0
↓
L1
↓
...
↓
Ln
```

查询时必须：

* 逐层查找
* 可能访问多个 SSTable
* 若 Key 不存在，需遍历所有层

导致：

> 1 次逻辑查询 → 多次 I/O

---

## 优化方式

* Bloom Filter
* Hash Index
* 减少无效磁盘访问

---

# 3️⃣ 写放大（Write Amplification）

写放大：

> 一次逻辑写入 → 多次磁盘重写

---

## 原因：Compaction

当：

* L0 写满
* 与 L1 合并
* 触发级联合并

可能导致：

* 数据被多次重写
* 大量磁盘 I/O

例如：

```
一次写入
→ L0
→ L1 合并
→ L2 合并
→ L3 合并
```

最终实际写入次数远大于 1。

# 📌 三、Pinecone 向量数据库：乘积量化（PQ）

向量数据库核心问题：

* 高维向量存储空间大
* 相似度计算开销大

例如：

```
768 维向量 × 1亿条数据
```

存储和计算压力巨大。

---

# 1️⃣ PQ 要解决的问题

## ① 存储空间过大

每个向量：

```
768 × 4 bytes = 3072 bytes
```

百万级数据即占数 GB。

---

## ② 计算复杂度高

相似度计算：

```
O(d)
```

大规模搜索非常耗时。

---

# 2️⃣ Product Quantization 原理

PQ 核心思想：

> 将高维向量拆分 + 聚类编码 + 近似计算

---

## Step 1：向量切分

例如：

```
784 维 → 28 个 28 维子向量
```

---

## Step 2：子空间聚类

对每个子向量空间：

* 聚成 256 类
* 每类一个中心

---

## Step 3：编码

不存原始浮点数：

* 只存“子空间类别 ID”
* 每个 ID 1 字节

原始：

```
28 × 28 × 4 bytes
```

压缩后：

```
28 bytes
```

存储量大幅降低。

---

## Step 4：查询时近似计算

查询向量：

* 同样分块
* 查表计算距离
* 使用查找表替代浮点运算

---

# 📊 PQ 的效果

| 原始向量 | PQ编码 |
| ---- | ---- |
| 高精度  | 近似   |
| 大存储  | 小存储  |
| 高计算  | 查表计算 |

属于：

> 空间换时间 + 精度换效率
